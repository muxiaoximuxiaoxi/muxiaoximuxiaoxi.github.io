UT平台驱动测试用例开发及使用指导

设计思路

1. 编写adc_test.c以及adc_test.h

在adc_test.h中声明结构体AdcTester，AdcTestConfig以及枚举类型AdcTestCmd、AdcTestConfig用于保存测试套的配置参数，当用一套测试套基于不同硬件测试一个模块时，若所需要的参数不同，则需要声明该结构体并在adc_driver_test.c中解析hcs中参数放入该结构体中，AdcTester继承父类AdcTestConfig并包含DevHandle类型的句柄，用于调用AdcOpen（）时获取ADC句柄，测试套中的每一个测试用例都对应AdcTestCmd中的一个枚举，通过枚举的cmd即可执行对应测试用例。

adc_test.c中编写测试用例，例如AdcTestSetupAll（）、AdcTestTearDownAll（）测试模块功能的AdcTestRead（）测试可靠性的AdcTestReliability（）以及测试模块的线程性能得AdcTestMultiThread（）用例等。

在adc_test.c中还要实例化一个全局的函数列表g_engry[],将AdcTestCmd的枚举与测试用例一一对应，以便于AdcTestExecute（）函数调用。

Static struct AdcTestEntry g_entry[] = {
{ADC_TEST_CMD_READ,AdcTestRead,” AdcTestRead”},
{ADC_TEST_CMD_MULTI_THREAD,AdcTestMultiThread,” AdcTestMultiThread”},
{ADC_TEST_CMD_RELIABILITY,AdcTestReliability,” AdcTestReliability”},
{ADC_TEST_CMD_SET_ALL,AdcTestSetUpAll,” AdcTestSetUpAll”},
{ADC_TEST_CMD_TEARDOWN_ALL,AdcTestTearDownAll,” AdcTestTearDownAll”}
}

2. 编写hdf_adc_entry_test.c以及hdf_adc_entry_test.h

在hdf_adc_entry_test.c和hdf_adc_entry_test.h中主要声明和定义了用户态调用内核态测试用例入口。

int32_t HdfAdcTestEntry(HdfTestMsg *msg)
{
    HDF_LOGE("%s: enter", __func__);
    if (msg == NULL) {
        return HDF_FAILURE;
    }
    msg->result = AdcTestExecute(msg->subCmd);
    return HDF_SUCCESS;
}

3. 编写用户态测试程序 hdf_adc_test.cpp

3.1 按照测试用例目录规划，定义测试套文件

测试套文件需要继承testing::Test类，命名以测试模块+test.cpp进行命名，来定义测试套文件。因此，开发一个外设用户态测试用例，可参考hdf_adc_test.cpp实现，新建hdf_xxx_test.cpp,实现自己模块的测试套。

3.2 定义测试套

实现测试套时需要先进行预处理和后处理操作（如下示例所示），（具体实现方法可参照hdf_adc_test.cpp文件），SetUpTestCase是针对该测试套下所有的用例执行前的预置处理；TearDownTestCase是针对该测试套下所有的用例执行后释放。

3.3 实现测试套具体的测试用例(包括用例注释和用例的逻辑实现，针对被测对象的特性编写测试用例，以hdf_adc_test.cpp文件为例说明)

逻辑实现：调用测试套HWTEST_F编写测试用例（Gtest框架中常用的测试套种类及功能请参考测试用例开发指导的开发指导篇章中的接口说明章节），测试用例的预期结果必须要有对应的断言，来判断测试结果是否与预期结果一致，如ASSERT_NE、EXPECT_EQ。HWTEST_F测试套编写的用例具备独立性，执行过程相互之间不受影响。

本例使用HWTEST_F宏，由于目前ADC框架仅支持内核态API,因此在用户态下进行测试，就要用到测试框架提供的服务，通过服务向内核发送msg，msg在hdf_adc_entry_test.c中被解析并调用对应的测试为例，测试内核态API的如下。

HWTEST_F(HdfliteAdcTest,AdcTestRead001,TestSize.Level1){
Struct HdfTestMsg msg={TEST_PAL_ADC_TYPE,ADC_TEST_CMD_READ,-1};
EXPECT_EQ(0,HdfTestSendMsgToService(&msg)); 
}

若待测试框架支持用户态API，则该测试程序中要分别对用户态和内核态进行测试，示例如下：

HWTEST_F(HdfliteAdcTest,AdcTestRead001,TestSize.Level1){
Struct HdfTestMsg msg={TEST_PAL_ADC_TYPE,ADC_TEST_CMD_READ,-1};
EXPECT_EQ(0,HdfTestSendMsgToService(&msg)); 
printf(“%s:kernel test done,then for user\n”,__func__);
EXPECT_EQ(0,AdcTestExecute(ADC_TEST_CMD_READ));
Printf(“%s exit\n”,__func__);   
}

5. 解析测试用例配置参数（可选）

若测试用例基于不同硬件执行时需要不同的参数，则需要将这些易变的参数放在hcs中，并通过adc_driver_test.c解析。

5.1 新增adc_test_config.hcs文件，写入易变参数：

 root{
    module = "adc";
    adc_config{
        adc_controller_tester{
            match_attr = "ADC_TEST";
            devid = 0;
            channel = 1;
            maxchannel = 1;
            activebit = 10;
            rate = 20000;
        }
    }
}

5.2 编写adc_driver_test.c文件，用于解析这些配置参数，入口如下：

struct HdfDriverEntry g_adcTestEntry = {
    .moduleVersion = 1,
    .Bind = AdcTestBind,
    .Init = AdcTestInit,
    .Release = AdcTestRelease,
    .moduleName = "PLATFORM_ADC_TEST",
};
HDF_INIT(g_adcTestEntry);

可以在AdcTestBind()或者AdcTestRelease()中解析配置参数。

编写一个AdcTestDisPatch()函数，并将其挂至service.Dispatch钩子上，这样既可在其他程序中调用AdcTestDisPatch()获取配置参数。

static int32_t AdcTestDispatch(struct HdfDeviceIoClient *client, int cmd, struct HdfSBuf *data, struct HdfSBuf *reply)
{
    HDF_LOGD("%s: enter!", __func__);

    if (cmd == 0) {
        if (reply == NULL) {
            HDF_LOGE("%s: reply is null!", __func__);
            return HDF_ERR_INVALID_PARAM;
        }
        if (!HdfSbufWriteBuffer(reply, &g_config, sizeof(g_config))) {
            HDF_LOGE("%s: write reply failed", __func__);
            return HDF_ERR_IO;
        }
    } else {
        return HDF_ERR_NOT_SUPPORT;
    }

    return HDF_SUCCESS;
}

在AdcTestBind()中挂钩子：

Service.Dispatch = AdcTestDisPatch;
Device->service = &service;

5.3 在device_info.hcs内添加用于解析参数的设备服务节点

设备服务节点添加有两种方式，第一种是新建一个sample_host节点, 另一种方式在原有源码配置plaform节点中添加(此时添加认为新增的驱动跟platform中的设备是同类设备)我们采用的是后者的方式。

root{
     device_info{
         test_host::host{\\\\
             hostName = "test_host";
             priority = 180;
             device_hdf_test::device{
                 //...
             }
             device_platform_test::device{
                 //...
                 /**********以下为新增内容************/
                 adc_tester::deviceNode{
                    policy = 2;
                    priority = 110;
                    preload = 0;
                    permission = 0644;
                    moduleName = "PLATFORM_ADC_TEST";
                    devicematchAttr = "ADC_TEST";
                    ServiceName = "ADC_TEST";
                 }
                 /**********以上为新增内容************/
             }
         }
     }
 }

其中”policy”字段为2，对外发布服务，” priority” 驱动启动优先级（0-200），值越大优先级越低, 优先级相同则不保证device的加载顺序,”moduleName”字段与adc_driver_test.c中的moduleName保持一致，” devicematchAtr”与adc_test_config.hcs中的”match_attr”保持一致。

8.挂载nfs

a)安装NFS服务器，右键以管理员身份运行：

做如下设置：

b)NFS注册

点击注册输入序列号及姓名

c) 记事本输入以下内容并Ctrl+S保存

D:\nfs -public -name:nfs （以个人创建nfs文件所在盘为准，注意-要手改防止-的格式被修改）

d) 点击win，找到“重启所有服务”
e) 右键打开文件所在位置，右键以管理员身份运行：

在D盘根目录创建nfs文件夹：

f) 开发板烧录镜像，上电开机，使用网线将PC和开发板连接，打开cmd，输入ipconfig查看开发板IP：（如：192.168.82.11）
g) 开发板使用ifconfig指令设置IP，掩码等，创建nfs目录，并通过mount指令挂载nfs服务器，指令见下方（开发板挂载指令）

注意：ifconfig eth0 指令设置的IP必须与PC上显示的IP处于同一网段，（如：192.168.82.12）。若挂载失败，需关闭PC防火墙后重新挂载。

h) 将测试用例的bin文件放在D:\nfs文件夹内

以下启动参数和开发板挂载命令复制后注意

LiteOS启动参数设置：

setenv bootargs consloe=ttyAMA0,115200n8 root=emmc fstype=vfat rootaddr=10M rootsize=21M rw;saveenv

setenv bootcmd fs probe 0;mmc read 0x0 0x80000000 0x800 0x4800;go 0x80000000;mw 0x12090000 0x001001e0;go 0x80000000;saveenv

LiteOS开发板挂载指令：

ifconfig eth0 169.254.83.11 netmask 255.255.0.0 gateway 169.254.1.1

mkdir nfs

mount 169.254.83.12:/nfs /nfs nfs

Linux启动参数设置：

setenv bootargs ‘mem=128M console=ttyAMA0,115200 root=/dev/mmcblk0p3 rw rootfstype=ext4 rootwait blkdevparts=mmcblk0:1M(boot),9M(kernel),50M(rootfs),50M(userfs)’

setenv bootcmd “mmc read 0x0 0x82000000 0x800 0x4800;bootm 0x82000000”; save; reset;

Linux开发板挂载指令：

ifconfig eth0 169.254.83.11 netmask 255.255.0.0

mkdir -p /nfs

mount -t nfs -o nolock,addr=169.254.83.12 169.254.83.12:/nfs /nfs

i) 开发板输入cd nfs进入nfs文件夹，ls即可找到测试用例文件，给测试用例权限 chmod 777 测试用例文件名（如可直接执行文件可忽略此步）
j) 执行测试用例
