PVE安装

Proxmox VE 7.4 ISO

创建虚拟机的基本信息

安装PVE

选择红框中的选项，敲回车键，安装PVE

选择，同意协议

选择系统盘

一般选择默认盘即可

配置国家和地区信息

国家选择“China”，时区选择“亚洲/上海”

配置密码和邮箱信息

这里的密码是“123456”权限的密码，邮箱只需要是合法的都没问题，不需要验证邮箱。

配置网卡、主机名、IP地址、DNS等信息

其中，选择默认网卡。主机名称“root1”。IP地址和DNS等信息可配置也可不配置，留待稍后配置也可以，但是最好是配置IP地址，如果需要修改，可以使用Web UI进行修改，这样比较简单。

检查核对信息，确定无误后点击安装

安装结束之后，出现界面证明安装成功。默认用户名为root，密码为安装时设置的密码。进入之后使用 ip addr 查看当前ip地址

通过浏览器访问 ip:8006，即可进入PVE的web页面

输入用户、密码并选择页面语言，即可登录至web页面

root1—>Ceph—>配置

CRUSH map 是 Ceph 存储集群的一种数据分布和存储方式，它有四个主要的部分：rules 规则、devices 设备、buckets（桶，也就是节点）、 types 类型

crush map一般ceph都会根据用户增加配置之后，自动生成，当然用户也可以先自主配置，然后根据crush map的配置在去添加相应配置

以下是一个crush map 的逐行解读

# Tunable：这些是 CRUSH 算法的一些调整参数，改变这些值可以微调数据分布
# 但一般情况下，不需要更改，当然如有需要，用户可自行更改
# begin crush map
tunable choose_local_tries 0
tunable choose_local_fallback_tries 0
tunable choose_total_tries 50
tunable chooseleaf_descend_once 1
tunable chooseleaf_vary_r 1
tunable chooseleaf_stable 1
tunable straw_calc_version 1
tunable allowed_bucket_algs 54


# Devices：这部分列出了集群的所有存储设备
# 本例 map 上有9个硬盘设备（osd.0 到 osd.8），都被分类在“hdd”类别下面
# devices
device 0 osd.0 class hdd
device 1 osd.1 class hdd
device 2 osd.2 class hdd
device 3 osd.3 class hdd
device 4 osd.4 class hdd
device 5 osd.5 class hdd
device 6 osd.6 class hdd
device 7 osd.7 class hdd
device 8 osd.8 class hdd


# Types：这是一个对象类型的层级列表。
# 本例列表里包含了从 osd 到 root 的所有对象类型。
# types
type 0 osd
type 1 host
type 2 chassis
type 3 rack
type 4 row
type 5 pdu
type 6 pod
type 7 room
type 8 datacenter
type 9 zone
type 10 region
type 11 root


# Buckets：这部分描述了你的存储设备是如何组织的。
# 本例 map 上有三个主机（vds2, vds3, vds4），每个主机有三个设备
# 每个主机和总体的 root 都被设置了权重，并使用了 straw2 算法来平衡存储负载

# buckets
host vds3 {
    id -3        # do not change unnecessarily
    # 这个存储桶的ID是-3。ID通常用于唯一标识一个存储桶，这里标记为“不要随意更改
    id -4 class hdd        # do not change unnecessarily
    # ID是-4，并且这个存储桶被标记为hdd类别。表示这个存储桶包含的是HDD（机械硬盘）类型的存储设备
    # weight 5.807
    # 这里有一个被注释掉的权重值，表示这个存储桶的权重原为5.807。权重用于在数据分布时确定存储桶的相对优先级
    # osd 的权重 ceph会根据磁盘自动生成，当然用户也可以自己设置
    alg straw2
    hash 0    # rjenkins1 
    # 使用hash函数0，注释表示这是rjenkins1哈希函数。哈希函数用于将数据键转化为存储桶的索
    item osd.2 weight 1.936
    # 在这个存储桶中，有一个名为osd.2的项，其权重为1.936。这表示在数据分布策略中，这个项（可能是一个OSD，即对象存储设备）具有相对权重
    item osd.0 weight 1.936
    # 在这个存储桶中，有一个名为osd.0的项，其权重为1.936。这表示在数据分布策略中，这个项（可能是一个OSD，即对象存储设备）具有相对权重
    item osd.1 weight 1.936
    # 在这个存储桶中，有一个名为osd.1的项，其权重为1.936。这表示在数据分布策略中，这个项（可能是一个OSD，即对象存储设备）具有相对权重
}
host vds4 {
    id -5        # do not change unnecessarily
    id -6 class hdd        # do not change unnecessarily
    # weight 5.807
    alg straw2
    hash 0    # rjenkins1
    item osd.4 weight 1.936
    item osd.5 weight 1.936
    item osd.3 weight 1.936
}
host vds2 {
    id -7        # do not change unnecessarily
    id -8 class hdd        # do not change unnecessarily
    # weight 5.807
    alg straw2
    hash 0    # rjenkins1
    item osd.6 weight 1.936
    item osd.8 weight 1.936
    item osd.7 weight 1.936
}
root default {
    id -1        # do not change unnecessarily
    id -2 class hdd        # do not change unnecessarily
    # weight 17.422
    alg straw2
    hash 0    # rjenkins1
    item vds3 weight 5.807
    item vds4 weight 5.807
    item vds2 weight 5.807
}


# Rules：这些规则描述了数据应该如何在集群中分布。
# 本例的四条规则都是复制类型的，分别决定了不同的存储和复制策略：
#     replicated_rule：允许数据在所有 osd 类型设备中选择复制
#     cluster_rule： 允许数据在主机级别复制
#     ssd_db_rule： 只在 vds4 的 osd 类型设备中复制数据
#     hdd_rule： 只在 vds3 的 osd 类型设备中复制数据


# rules
rule replicated_rule {
    id 0
    # 给这个规则分配一个唯一的ID，这里是0
    type replicated
    # 指定这是一个复制类型的规则，即数据会被复制到多个节点上以提供冗余和可用性
    min_size 1
    # 设置最小的副本数量为1，即至少有一个副本
    max_size 10
    # 设置最大的副本数量为10，即最多可以有10个副本
    step take default
    # 这是第一个步骤，take default意味着使用默认的权重和优先级来选择存储节点
    step chooseleaf firstn 0 type osd
    # 这是第二个步骤，用于选择一个叶子节点来放置副本。这里使用firstn 0来选择第一个匹配的叶子节点，类型为OSD
    step emit
    # 这是最后一个步骤，表示规则执行完成，可以放置副本了
}
rule cluster_rule {
    id 1
    type replicated
    min_size 1
    max_size 10
    step take default
    step chooseleaf firstn 0 type host
    step emit
}
rule ssd_db_rule {
    id 2
    type replicated
    min_size 1
    max_size 10
    step take vds4
    step choose firstn 0 type osd
    step emit
}
rule hdd_rule {
    id 3
    type replicated
    min_size 1
    max_size 10
    step take vds3
    step choose firstn 0 type osd
    step emit
}

# end crush map
  1. 先创建 pool 选择压缩（带 C 为压缩，不带C为不压缩）
  2. 选择rule，设置最大、最小副本数
  3. 设置rule的step 具体规则中如何去执行

关于上述的 buckets 算法补充

目前有 5 种算法来实现节点的选择， 包括 Uniform、 List、 Tree、 Straw、 Straw2

Uniform:
简单随机选择。每个节点被选中的概率是相同的。
优点：简单、公平。
缺点：在大规模数据集上可能效率不高。


List:
按照某种顺序（如顺序访问）选择节点。
优点：实现简单，适用于顺序处理或某些特定场景。
缺点：缺乏随机性可能导致某些节点过度使用。


Tree:
基于某种树形结构（如B树或红黑树）选择节点。
优点：树形结构有助于减少I/O操作，提高查询效率。
缺点：实现复杂度较高，并且不适用于所有场景。


Straw:
基于概率的随机选择，给每个节点分配一个权重，根据权重随机选择节点。
优点：能够根据节点的性能或其他属性进行选择性访问，提高效率。
缺点：需要维护节点的权重，实现稍复杂。


Straw2:
Straw算法的改进版本，可能考虑了更多的因素或优化了某些部分。
优点：在某些情况下可能比Straw算法更高效。
缺点：实现可能更复杂，具体取决于与Straw相比做了哪些改进。
每种算法都有其适用的场景和限制，选择哪种算法取决于具体的需求和场景。在实际应用中，通常会根据数据的规模、结构、访问模式以及性能要求等因素来选择最合适的算法。

关于上述的 rule 算法补充

Replicated rule：这是最常见的规则，它将数据复制在不同的设备或组（例如主机，机架，数据中心等）上以提供容错能力

Erasure coding rule：Erasure coding 是一种数据保护技术，可以在损失一定数量的 OSD 时仍然能恢复数据。这个规则需要更复杂的设置，并需要你定义如何分割和分发各个数据的片段

Tiering rule：这个规则类型用于管理多层存储系统。比如，你可能会创建一个规则，将热数据存储在 SSD 上，然后将冷数据迁移到硬盘驱动器上

Device class rule：这个规则允许你根据存储设备的类型（例如 SSD、HDD）来定制数据的分布模式

在创建磁盘时，数据库磁盘可选择为当前创建磁盘，也可选择为其他可选的ssd可用分区（具体分区设置可通过linux指令进行创建）

选择当前磁盘，表示安装当前磁盘的格式进行读写数据，即当前磁盘为hdd就按hdd的方式读写，为ssd就按ssd的方式读写
选择其他分区，可实现使用sdd可用分区给当前的hdd磁盘读取数据，即相当于给hdd赋予ssd的读写速率

OSD status
up 表示 该磁盘为运行状态或者可用状态
down 表示 该磁盘为未运行状态或者不可用状态
in 表示 在ceph中
out 表示 在ceph外

集群删除节点

1、查看当前节点信息
ls /etc/pve/nodes 

2、停止节点上的corosync和pve集群服务

systemctl stop pve-cluster
systemctl stop corosync


3、在本地模式下再次启动群集文件系统

pmxcfs -l

4、删除corosync配置文件

rm /etc/pve/corosync.conf
rm -r /etc/corosync/*

5、将文件系统作为正常服务重新启动

killall pmxcfs
systemctl start pve-cluster

此时刷新删除节点的web页面，会发现集群信息

6、节点现在与集群分离。可以使用以下命令将其从集群的任何剩余节点中删除

pvecm delnode oldNodeName


7、如果由于剩余节点中的仲裁丢失而导致命令失败，可以将 expected 设置为1作为解决方法，然后重复pvecm-delnode命令。

pvecm expected 1
pvecm delnode oldNodeName
 
 
 8、现在切换回分离的节点并删除其上所有剩余的集群文件。这确保了该节点可以毫无问题地再次添加到另一个集群中。如果其他节点的配置文件仍在集群文件系统中，那么可能也需要清理这些文件。在确保具有正确的节点名称后，您可以简单地从 /etc/pve/nodes/NODENAME 递归地删除整个目录。

rm /var/lib/corosync/*
ls /etc/pve/nodes  # 打印出节点名称
pvecm delnode nodeName
rm -rf /etc/pve/nodes/nodeName
